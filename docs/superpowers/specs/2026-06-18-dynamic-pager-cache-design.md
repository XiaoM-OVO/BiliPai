# 动态分页缓存与滑动性能设计

## 目标

修复动态页左右滑动切换后手势迟滞、列表被回顶，以及相邻页面复用当前 Tab 内容导致显示旧数据或不更新的问题。每个动态 Tab 保留独立滚动位置，并在切页后立即展示该 Tab 的内存缓存。

## 根因

- Pager 滑动期间在组合阶段读取 `currentPage` 和 `isScrollInProgress`，扩大了高频重组范围。
- `derivedStateOf` 和页面组合中通过 `getOrPut` 写入快照状态，形成组合阶段反向写入。
- 所有 Pager 页面共用当前选中 Tab 的 `DynamicUiState.items` 和过滤结果，相邻页会显示错误内容。
- 切页完成后强制执行 `scrollToItem(0)`，破坏独立滚动位置，并增加主线程工作。
- `beyondViewportPageCount = 1` 会提前组合包含图片卡片的重列表，增加滑动负担。
- ViewModel 只有一份时间线状态和全局请求锁，切换 Tab 时新请求会覆盖旧 Tab 状态，旧请求也无法更新自己的缓存。

## 状态设计

新增按请求类型索引的时间线页面状态。`all`、`video`、`pgc`、`article` 各自保存：

- 动态列表
- 首次加载状态
- 错误信息与错误来源
- 是否可继续分页
- 增量刷新边界与新增数量

`DynamicUiState` 对外暴露不可变页面状态映射。当前选中 Tab 仍由现有 `selectedTab` 流维护，UI 根据每个 Pager 页对应的逻辑 Tab 读取页面状态，不再让所有页共享当前页数据。

UP 专属动态继续使用现有 `userItems`、`userIsLoading`、`userError` 和 `hasUserMore`，本次不增加“每个 UP 一份缓存”。磁盘缓存继续只恢复和保存“全部”Tab，避免改变持久化格式及迁移逻辑；其他 Tab 仅保留进程内缓存。

## 请求与刷新

- 切页稳定后才调用 `setSelectedTab`，切页过程中不发起请求。
- 目标 Tab 有缓存时立即展示缓存，并在后台刷新；刷新期间保留旧内容。
- 目标 Tab 无缓存时显示该 Tab 自己的首次加载状态。
- 加载锁和请求令牌按请求类型隔离。快速连续切页时，请求结果只更新其所属页面状态，不能覆盖其他 Tab。
- 分页继续使用 Repository 现有按 `DynamicFeedScope` 和请求类型隔离的游标。
- 手动下拉刷新和加载更多明确携带当前逻辑 Tab，避免异步执行时读取已经变化的全局选中项。
- 请求失败只更新所属 Tab；已有缓存时保留内容并显示刷新错误，首次加载失败时显示空态错误。

## Compose 设计

- 使用普通 `remember` 为可见 Tab 创建稳定的 `LazyStaggeredGridState` 映射，组合读取路径不写快照状态。
- 活跃列表状态由稳定页索引查表获得；不在 `derivedStateOf` 中创建列表状态。
- 使用 `snapshotFlow { pagerState.settledPage }` 和 `distinctUntilChanged()` 同步 ViewModel，只在落页后提交选择。
- 顶部 Tab 点击只驱动 Pager；ViewModel 选择由统一的落页同步处理，避免重复动画和重复请求。
- 每个 Pager 页面按自己的逻辑 Tab 读取页面状态并计算可见内容。
- 移除切页后的强制回顶和 `beyondViewportPageCount = 1`，保留每个 Tab 的滚动位置。
- 顶栏继续显示当前可见 Pager 页；不新增视觉样式或动画。

## 测试

先增加失败测试，再实施生产代码：

- 页面状态按请求类型独立合并、刷新和失败，不互相覆盖。
- 不同请求类型的令牌独立判定，较早页面请求可以更新自己的缓存。
- Pager 只在稳定页变化后提交逻辑 Tab。
- 每个页面读取对应 Tab 的状态和过滤结果。
- 结构测试确认不再在 `derivedStateOf` 中写列表状态、不强制切页回顶、不预组装重页面。

完成后运行动态功能定向单测和 `:app:compileDebugKotlin`。若设备已有可运行版本且登录态可用，再通过 ADB 检查切页、内容更新和滚动位置；不执行 APK 打包、安装或 Release 构建。

## 非目标

- 不改变动态接口、Repository 分页协议或磁盘缓存格式。
- 不新增依赖。
- 不重构 UP 专属动态为按用户缓存。
- 不调整动态页视觉设计、卡片实现或图片加载策略。

## 风险控制

- 状态映射会增加进程内动态对象引用，限定为四个时间线 Tab，且复用现有分页数量上限。
- 点赞、删除等跨缓存更新必须同步修改所有包含目标动态的页面，避免同一动态在不同 Tab 显示不一致。
- 关注用户列表继续从“全部”时间线与现有关注接口构建，避免分类 Tab 覆盖侧栏来源。
